Microsoft Graph permissions

https://docs.microsoft.com/en-us/graph/permissions-reference

Microsoft Graph permission names follow a simple pattern: resource.operation.constraint. For example, User.Read grants permission to read the profile of the signed-in user, User.ReadWrite grants permission to read and modify the profile of the signed-in user, and Mail.Send grants permission to send mail on behalf of the signed-in user.

The constraint element of the name determines the potential extent of access your app will have within the directory. Currently Microsoft Graph supports the following constraints:

  • All grants permission for the app to perform the operations on all of the resources of the specified type in a directory. For example, User.Read.All potentially grants the app privileges to read the profiles of all of the users in a directory.
  • Shared grants permission for the app to perform the operations on resources that other users have shared with the signed-in user. This constraint is mainly used with Outlook resources like mail, calendars, and contacts. For example, Mail.Read.Shared, grants privileges to read mail in the mailbox of the signed-in user as well as mail in mailboxes that other users in the organization have shared with the signed-in user.
  • AppFolder grants permission for the app to read and write files in a dedicated folder in OneDrive. This constraint is only exposed on Files permissions and is only valid for Microsoft accounts. -If no constraint is specified the app is limited to performing the operations on the resources owned by the signed-in user. For example, User.Read grants privileges to read the profile of the signed-in user only, and Mail.Read grants permission to read only mail in the mailbox of the signed-in user.

Note: In delegated scenarios, the effective permissions granted to your app may be constrained by the privileges of the signed-in user in the organization.

Get access without a user

Some apps call Microsoft Graph with their own identity and not on behalf of a user. In many cases, these apps are background services or daemons that run on a server without the presence of a signed-in user. An example of such an app might be an email archival service that wakes up and runs overnight. In some cases, apps that have a signed-in user present may also need to call Microsoft Graph under their own identity. For example, an app may need to use functionality that requires more elevated privileges in an organization than the signed-in user may have.

Apps that call Microsoft Graph with their own identity use the OAuth 2.0 client credentials grant flow to get access tokens from Azure AD. This describes the basic steps to configure a service and use the OAuth client credentials grant flow to get an access token.

  • Register your app.
  • Configure permissions for Microsoft Graph on your app.
  • Get administrator consent.
  • Get an access token.
  • Use the access token to call Microsoft Graph.
  • Setup Application Access Policy.

Register your app

You must first register your app at the Azure Active Directory app registration portal.

Click on the new registration name.

Click Add a permission button.

Click the large Microsoft Graph button.

Click the Application permissions button.

Select Mail.Read and Mail.Send checkboxes.

Click the Add permissions button.

Click Grant admin consent.

Repeat for SharePoint permissions.

Certificates & secrets

Azure Active Directory (Azure AD) supports two types of authentication: password-based authentication (client secret) and certificate-based authentication. It’s recommended that your application use a certificate.

Client Secret

To create a client secret, click the button. Enter a single line description. Select the secret lifespan. Click the Add button.

Copy the secret value now. You cannot view the secret value after leaving the webpage.

X509 certificate

Select Certificates & secrets from left menu. Click Upload Certificate button.

Create a self-signed public certificate

https://docs.microsoft.com/en-us/azure/active-directory/develop/howto-create-self-signed-certificate

Using PowerShell:

Use a self-signed certificate only for Development or Testing, not Production.

$certname = "{certificateName}"    ## Replace {certificateName}

$cert = New-SelfSignedCertificate -Subject "CN=$certname" -CertStoreLocation "Cert:\CurrentUser\My" -KeyExportPolicy Exportable -KeySpec Signature -KeyLength 2048 -KeyAlgorithm RSA -HashAlgorithm SHA256

Export-Certificate -Cert $cert -FilePath "C:\Users\yourusername\Desktop\$certname.cer"   ## Specify your preferred location

Scoping application permissions to specific Exchange Online mailboxes

In the images below, replace whirata.onmicrosoft.com with hawaii.gov.

https://docs.microsoft.com/en-us/graph/auth-limit-mailbox-access

Administrators who want to limit the app access to a specific set of mailboxes can use the New-ApplicationAccessPolicy PowerShell cmdlet to configure access control.

  • Get the app’s Application (client) ID from the Azure app registration.
  • Create a mail-enabled security group.
  • Create an application access policy.

Identify the app’s Application (client) ID

Open the Azure Active Directory portal. Go to your app’s App registration. The Application ID is listed near the top of the overview.

https://aad.portal.azure.com/#blade/Microsoft_AAD_RegisteredApps/ApplicationMenuBlade/Overview/appId/900f9389-477a-4f71-878e-87ffa10f6e82/isMSAApp/

Setup Exchange Mail-Enabled Security Group

  • Open the Exchange Admin Center.
  • Select the Groups option.
  • Click the Add a group button.
  • Choose group type Mail-enabled security.
  • Enter a name for your group. Enter a description.
  • Enter an email address. Check the Require owner approval to join the group checkbox.
  • Click the Create group button.
  • Click the Close button.
  • The new group is now in the Group listing.
  • Select the new group and add new member test.govazure@hawaii.gov.

Enter a name for your group. Enter a description.

Enter an email address. Check the Require owner approval to join the group checkbox.

Click the Create group button.

Click the Close button.

Create an Application Access Policy using PowerShell

You can get a listing of your Application Access Policies.

https://docs.microsoft.com/en-us/powershell/module/exchange/get-applicationaccesspolicy

  • Open PowerShell.
  • Import the Exchange PowerShell module.
  • Connect to Exchange with modern authentication.
  • Create new Application Access Policy.
  • Test the policy with various email accounts. The AccessCheckResult will be Granted or Denied.
  • PowerShell error message, click here.
PS> Import-Module ExchangeOnlineManagement
PS> Connect-ExchangeOnline -UserPrincipalName yourname@hawaii.gov
PS> Get-ApplicationAccessPolicy
PS> New-ApplicationAccessPolicy -AppId 900f9389-477a-4f71-878e-87ffa10f6e82 -PolicyScopeGroupId govcrmtestsecuritygroup@whirata.onmicrosoft.com -AccessRight RestrictAccess -Description "Restrict this app to members of distribution group GovCRM_TEST_MailSecurityGroup."
PS> Test-ApplicationAccessPolicy -Identity graphtest@whirata.onmicrosoft.com -AppId 900f9389-477a-4f71-878e-87ffa10f6e82
PS> Test-ApplicationAccessPolicy -Identity whirata@whirata.onmicrosoft.com -AppId 900f9389-477a-4f71-878e-87ffa10f6e82
